引入规范的意义

  1. 降低新成员融入团队的成本, 一定程度是上避免挖坑.
  2. 提高开发效率、团队协作效率, 降低沟通成本.
  3. 实现高度统一的代码风格, 方便review, 提高项目的可维护性.
  4. 规范是实现自动化的基础
  5. 规范是一个团队知识沉淀的直接产出

1. 工作流规范

1.1 开发
1.1.1 版本规范
`语义化版本`
+ 主版本号: 做了不兼容的API修改
+ 次版本号: 做了向下兼容的功能性新增
+ 修订号: 做了向下兼容的问题修正
1.1.2 版本控制系统规范
比如分支规范、提交规范、merge规范等.
以现有项目的流程为例, 项目主要分支有两个, 分别为master(生产分支), dev(开发、测试).]
对于新需求研发,新建feature分支, 待开发完成后提交 MR to dev分支, 指定审核人员review代码后, dev分支为最新代码, 同时自动出发jenkins CI操作.
对于线上bug, 新建HotFix分支, 待问题修复后, 将该分支合并入dev分支, 待测试无误后提交 dev MR to master. 触发上线操作, 完成问题修复.
进入提测阶段:
1. 测试环境作为第一轮的测试base, 如该轮测试结束后, 进入第二轮测试, 提交dev MR to master.
2. 第二轮测试在pre_prd环境中进行, 该环境代码与生产环境代码一致.
上线:
1. 在当前master分支打上Tag
2. 在jenkins中填写Tag, 触发构建操作.
1.1.3 提交信息规范
可借鉴的提交规范, 如Angular的提交信息规范,Atom, Ember, Eslint, JQuery等. 或者借助检验工具, 比如conventiaonal-changelog, commitlin, commitizen, standard-changelog等.
